📘 CAPÍTULO 1: ¿QUÉ ES ANGULAR Y PARA QUÉ SIRVE?

Imaginá que querés construir una casa. Tenés dos opciones:

Opción A: Comprás ladrillos, cemento, arena, hierro, y construís todo desde cero. Vos decidís dónde va cada pared, cada ventana, cada caño. Tenés control total, pero cada detalle depende de vos.

Opción B: Usás un sistema de construcción prefabricado. Ya vienen los paneles, las vigas, el techo. Vos decidís cómo distribuís los ambientes y qué color pintás las paredes, pero la estructura base ya está resuelta.

Angular es la Opción B para aplicaciones web. Es un framework creado por Google que te da la estructura, las herramientas y las reglas para construir aplicaciones web modernas. Vos llenás los espacios con tu código.

💡 ¿Framework o Librería?
Una librería es como una caja de herramientas: vos decidís cuándo y cómo usar cada herramienta.
Un framework es como un kit de construcción: ya viene con las piezas y las instrucciones. Vos armás siguiendo el plano.

¿Qué tipo de aplicaciones se construyen con Angular?

Angular construye SPA (Single Page Applications): aplicaciones de una sola página. A diferencia de una web tradicional donde cada click recarga la página entera, en una SPA la página se carga una sola vez y después solo se intercambian datos con el servidor.

WEB TRADICIONAL SPA CON ANGULAR (UTNito) ──────────────── ──────────────────────── Usuario hace click en "Chat" Usuario hace click en "Chat" ↓ ↓ Navegador pide /chat al servidor Angular muestra el componente Chat ↓ ↓ Servidor devuelve TODO el HTML Angular pide SOLO los datos (JSON) ↓ ↓ Pantalla en blanco mientras carga La pantalla NUNCA se pone en blanco ↓ ↓ El navegador renderiza la página Angular actualiza solo lo que cambió

UTNito es una SPA. Cuando iniciás sesión y ves el chat, la página nunca se recarga. Angular solo pide los mensajes nuevos al backend y los muestra instantáneamente.

¿Qué necesito saber antes de Angular?

Angular usa TypeScript, que es JavaScript con tipos. Si ya leíste la guía de la Clase 11 del backend, ya sabés qué es TypeScript. Pero no te preocupes, voy a explicar cada cosa que aparezca.

¿Qué es un componente?

Un componente es un pedazo de pantalla que hace una cosa específica. Es como un ladrillo LEGO: cada pieza tiene su forma y función, y al unirlas construís la aplicación completa.

Ejemplo con UTNito: La pantalla del chat está formada por varios componentes:

Cada rectángulo con bordes dobles es un componente independiente. Cada uno tiene su propio archivo de lógica (TypeScript), su propia vista (HTML) y sus propios estilos (CSS).

🧭 RUTAS RÁPIDAS - PARA QUE VEAS TU PROYECTO FUNCIONANDO

Ya creaste tus primeros componentes (Login y Chat). Pero si abrís http://localhost:5300, no ves nada. ¿Por qué? Porque Angular no sabe cuándo mostrar cada componente. Necesitamos decirle: "si la URL es /login, mostrá el Login. Si es /chat, mostrá el Chat".

Esto se llama enrutamiento y se configura en 3 pasos simples. No te preocupes si no entendés todo ahora, en el Capítulo 14 lo vas a ver a fondo. Por ahora, solo necesitás que tu proyecto funcione.

¿CÓMO SABE ANGULAR QUÉ MOSTRAR? URL: /login → LoginComponent URL: /chat → ChatComponent URL: / → Redirige a /login

Los 3 archivos que necesitás tocar

Archivo¿Qué hace?
app.routes.tsDefine qué componente se muestra en cada URL
app.module.tsRegistra las rutas y declara los componentes
app.component.htmlPone el marcador donde Angular inserta el componente

🛠️ MANOS A LA OBRA

Objetivo: Configurar las rutas mínimas para que Login y Chat se vean en el navegador.

📋 Paso 1: Crear el archivo de rutas

🧠 PRIMERO PENSÁ: ¿Qué URLs necesitamos? Una para el login y otra para el chat.

Creá src/app/app.routes.ts con este contenido:

📄 app.routes.ts
import { Routes } from '@angular/router'; // Importamos los componentes que queremos mostrar import { ChatComponent } from './chat/chat.component'; import { LoginComponent } from './login/login.component'; // Cada ruta dice: "cuando la URL sea X, mostrá el componente Y" export const routes: Routes = [ { path: 'login', component: LoginComponent }, // /login → Login { path: 'chat', component: ChatComponent }, // /chat → Chat { path: '', redirectTo: '/login', pathMatch: 'full' }, // / → redirige a /login { path: '**', redirectTo: '/login' }, // Cualquier otra URL → /login ];

💡 Explicación rápida:
path: 'login' → la parte de la URL después del dominio (ej: /login)
component: LoginComponent → qué componente se muestra en esa URL
redirectTo: '/login' → cuando la URL está vacía, manda al login
path: '**' → comodín: cualquier URL que no exista, redirige al login

📋 Paso 2: Configurar el módulo principal

🧠 PRIMERO PENSÁ: ¿Dónde le decimos a Angular que use estas rutas y que existen nuestros componentes?

Abrí src/app/app.module.ts y reemplazá todo por esto:

📄 app.module.ts
import { NgModule } from '@angular/core'; import { BrowserModule } from '@angular/platform-browser'; import { RouterModule } from '@angular/router'; // └──────────┘ // Módulo que maneja las rutas import { AppComponent } from './app.component'; import { LoginComponent } from './login/login.component'; import { ChatComponent } from './chat/chat.component'; import { routes } from './app.routes'; // └────┘ // Las rutas que definimos en el paso 1 @NgModule({ declarations: [ AppComponent, LoginComponent, // ← Registramos el componente de login ChatComponent // ← Registramos el componente de chat ], imports: [ BrowserModule, RouterModule.forRoot(routes) // ← Registramos las rutas ], providers: [], bootstrap: [AppComponent], }) export class AppModule {}

💡 Explicación rápida:
declarations → le dice a Angular "estos componentes existen, son míos"
imports → trae módulos que necesitamos (RouterModule.forRoot(routes) activa las rutas)
bootstrap → el componente que arranca primero (siempre AppComponent)

📋 Paso 3: Poner el marcador de posición

🧠 PRIMERO PENSÁ: ¿Dónde va a poner Angular el componente según la URL?

Ya tenés tu componente creado. Ahora necesitás conectar lo que pasa en el TypeScript (el cerebro) con lo que se ve en el HTML (la cara). Esa conexión se llama binding.

Hay 4 tipos de binding en Angular. Hoy vamos a ver los 2 primeros: interpolación y property binding.

Interpolación {{ }} - Mostrar datos del TS en el HTML

La interpolación es la forma más simple de mostrar datos. Todo lo que está entre {{ }} se evalúa como una expresión TypeScript y el resultado se convierte en texto visible en la pantalla.

📄 Ejemplos de interpolación
<!-- Mostrar una variable --> <p>{{ username }}</p> <!-- Si username = 'carlos', el navegador ve: <p>carlos</p> --> <!-- Mostrar el resultado de una función --> <p>{{ getInitials() }}</p> <!-- Si getInitials() devuelve 'CG', el navegador ve: <p>CG</p> --> <!-- Mostrar una operación matemática --> <p>{{ 2 + 2 }}</p> <!-- El navegador ve: <p>4</p> --> <!-- Mostrar una propiedad de un objeto --> <p>{{ conversation.title }}</p> <!-- Si conversation = { title: 'Planificación' }, muestra: <p>Planificación</p> -->

Reglas de la interpolación:

✅ Se puede poner: variables, llamadas a funciones, operaciones matemáticas, propiedades de objetos

❌ No se puede poner: if, for, while, declaraciones de variables (let x = 5)

💡 Analogía: La interpolación es como un cartel digital. Lo que pongas entre {{ }} aparece en pantalla. Si cambia el valor en TypeScript, el cartel se actualiza solo.

Property Binding [ ] - Del TS al HTML

El property binding envía datos DESDE la variable TypeScript HACIA una propiedad del HTML. El flujo es: TS → HTML.

Mientras que la interpolación muestra texto, el property binding asigna valores a propiedades como disabled, src, href, value.

📄 Property Binding - Ejemplos
<!-- ─── SIN PROPERTY BINDING ─── --> <!-- El valor "carlos" es un TEXTO FIJO. Nunca cambia. --> <input value="carlos"> <!-- ─── CON PROPERTY BINDING ─── --> <!-- El valor viene de la variable username del .ts --> <!-- Si username cambia en el .ts, el input se actualiza automáticamente --> <input [value]="username"> <!-- └─────┘ └───────┘ Propiedad Variable del .ts HTML --> <!-- ─── MÁS EJEMPLOS ─── --> <!-- Deshabilitar un botón según una condición --> <button [disabled]="isLoading">Enviar</button> <!-- Si isLoading = true → botón deshabilitado --> <!-- Si isLoading = false → botón habilitado --> <!-- Cambiar la imagen según una variable --> <img [src]="userAvatar" [alt]="userName"> <!-- userAvatar = 'https://...foto.jpg' --> <!-- userName = 'Carlos Gardel' --> <!-- Pasar datos a un componente hijo --> <app-conversation-item [conversation]="selectedConversation"> <!-- └───────────┘ └───────────────────┘ Propiedad del Variable del padre componente hijo que contiene el dato -->
💡 Analogía: El property binding es como un control remoto. Vos cambiás el valor en TypeScript (el control) y el HTML (la tele) responde automáticamente.

Diferencia entre interpolación y property binding

Interpolación {{ }}Property Binding [ ]
Muestra texto en pantallaAsigna valores a propiedades HTML
Ej: {{ username }}Ej: [disabled]="isLoading"
Siempre es textoPuede ser boolean, string, objeto
Va entre etiquetasVa dentro de la etiqueta de apertura

🔘 CAPÍTULO 4: EVENT BINDING Y TWO-WAY BINDING

En el capítulo anterior aprendiste a mostrar datos en el HTML (interpolación) y a enviar datos del TypeScript al HTML (property binding). Ahora vamos a aprender lo contrario: escuchar lo que hace el usuario y responder.

Event Binding ( ) - Del HTML al TypeScript

El event binding escucha lo que hace el usuario (click, tecla, envío de formulario) y ejecuta funciones del TypeScript. El flujo es: HTML → TS.

📄 Event Binding - Tipos de eventos
<!-- ─── EVENTOS DE CLICK ─── --> <!-- Cuando el usuario hace click, ejecuta el método onLogin() --> <button (click)="onLogin()">Ingresar</button> <!-- └────┘ └───────┘ Evento Método del .ts --> <!-- ─── EVENTOS DE TECLADO ─── --> <!-- Cuando el usuario presiona Enter, ejecuta onSearch() --> <input (keyup.enter)="onSearch()"> <!-- └─────────┘ Evento: tecla Enter --> <!-- ─── EVENTOS DE FORMULARIO ─── --> <!-- Cuando el usuario envía el formulario (Enter o click en submit) --> <form (ngSubmit)="onSubmit()"> <!-- └──────┘ Evento de Angular para formularios (ngSubmit) previene la recarga de la página -->
Evento¿Cuándo se dispara?Ejemplo
(click)Click del mouse<button (click)="save()">
(input)Al escribir en un input<input (input)="onChange($event)">
(keyup.enter)Tecla Enter<input (keyup.enter)="search()">
(ngSubmit)Envío de formulario<form (ngSubmit)="onSubmit()">
(mouseenter)Mouse entra al elemento<div (mouseenter)="showTooltip()">
(change)Cambio en select/checkbox<select (change)="onSelect($event)">
💡 Analogía: El event binding es como un timbre. Vos configurás el timbre (el evento) para que cuando alguien lo presione (el usuario haga click), suene una campana (se ejecute la función).

Two-Way Binding [( )] - Sincronización bidireccional

Combina [ ] (property binding) y ( ) (event binding) en un solo símbolo. El dato fluye en ambas direcciones entre el HTML y el TypeScript.

📄 Two-Way Binding con [(ngModel)]
<!-- ─── TWO-WAY BINDING ─── --> <input [(ngModel)]="username"> <!-- └──────┘ Directiva que combina [value] y (input) --> <!-- Es EXACTAMENTE equivalente a: --> <input [value]="username" (input)="username = $event.target.value">

¿Cómo funciona paso a paso?

1. El usuario escribe "car" en el input 2. El evento (input) se dispara 3. $event.target.value es "car" 4. username = "car" (se actualiza la variable en el .ts) 5. Como username cambió, [value]="username" actualiza el input 6. El input muestra "car" Resultado: La variable y el input siempre tienen el mismo valor. Nunca están desincronizados.
⚠️ IMPORTANTE: Para usar [(ngModel)] en formularios template-driven, necesitás importar FormsModule en tu módulo. Pero en UTNito usamos formularios reactivos (que veremos en el Capítulo 5), así que no vamos a usar [(ngModel)] en este proyecto. Es importante que lo conozcas porque lo vas a ver en muchos tutoriales y proyectos.

¿Qué es un formulario reactivo?

Un formulario reactivo es un objeto TypeScript que representa todo el formulario. En lugar de poner validaciones en el HTML, las definís en el TypeScript. El HTML solo muestra los campos y errores.

┌─────────────────────────────────────────────────────────────┐ │ FORMULARIO REACTIVO │ │ │ │ TypeScript (.ts) HTML (.html) │ │ ┌────────────────────┐ ┌────────────────────────────┐ │ │ │ FormGroup │ │ <form [formGroup]="..."> │ │ │ │ ├─ FormControl │ ←→ │ <input formControlName │ │ │ │ │ (username) │ │ ="username"> │ │ │ │ ├─ FormControl │ ←→ │ <input formControlName │ │ │ │ │ (password) │ │ ="password"> │ │ │ │ └─ Validators │ │ <button>Login</button> │ │ │ └────────────────────┘ └────────────────────────────┘ │ │ │ │ La lógica vive acá Solo muestra y escucha │ └─────────────────────────────────────────────────────────────┘

Las piezas de un formulario reactivo

1. FormGroup: Es el formulario completo. Agrupa todos los campos.

2. FormControl: Es cada campo individual (username, password).

3. FormBuilder: Es una herramienta que nos ayuda a crear FormGroups y FormControls más fácilmente.

4. Validators: Son reglas que validan los datos (requerido, longitud mínima, etc.).

📄 Ejemplo de formulario reactivo
// 1. Inyectamos FormBuilder en el constructor constructor(private formBuilder: FormBuilder) { // 2. Creamos el FormGroup con FormBuilder this.loginForm = this.formBuilder.group({ // 3. Cada campo es un FormControl con validadores username: ['carlos.gardel', [Validators.required, Validators.minLength(3)]], // └──────────────┘ └─────────────────────────────────────┘ // Valor inicial Validadores (reglas) password: ['123456', [Validators.required, Validators.minLength(1)]] }); }
💡 Analogía: Un formulario reactivo es como un examen multiple choice. El profesor (TypeScript) define las preguntas y respuestas correctas (validadores). El alumno (HTML) solo muestra las preguntas y marca las respuestas. El profesor corrige automáticamente.

🏗️ CAPÍTULO 6: DIRECTIVAS *ngIf Y *ngFor

Hasta ahora nuestros componentes muestran contenido estático. Pero una aplicación real necesita mostrar y ocultar elementos según condiciones, y repetir elementos para cada item de una lista. Para eso existen las directivas estructurales.

Las directivas son instrucciones que le damos a Angular para que modifique el HTML dinámicamente. Las estructurales (con *) pueden crear o destruir elementos del DOM.

*ngIf - Mostrar u ocultar condicionalmente

*ngIf crea o destruye un elemento según una condición. Si la condición es true, el elemento existe. Si es false, el elemento no existe en absoluto (no está oculto, fue eliminado del DOM).

📄 *ngIf - Ejemplos
<!-- ─── ngIf BÁSICO ─── --> <!-- Solo muestra el mensaje de error si formMessage NO está vacío --> <p class="error-text" *ngIf="formMessage">{{ formMessage }}</p> <!-- ¿Cómo funciona? --> <!-- Si formMessage = 'Complete valid credentials...' --> <!-- <p class="error-text">Complete valid credentials...</p> ← Existe en el DOM --> <!-- Si formMessage = '' --> <!-- No hay <p>. NO EXISTE en el DOM. No está oculto, NO EXISTE. -->

Diferencia entre *ngIf y [hidden]:

📄 *ngIf vs [hidden]
<!-- Con *ngIf: el elemento NO EXISTE en el DOM --> <div *ngIf="false">No me ve nadie</div> <!-- El HTML resultante NO tiene este div --> <!-- Con [hidden]: el elemento EXISTE pero está OCULTO --> <div [hidden]="true">Estoy oculto pero existo</div> <!-- El HTML resultante: <div hidden>Estoy oculto pero existo</div> -->
SituaciónUsarMotivo
Elemento que rara vez se muestra*ngIfAhorra memoria, no existe en el DOM
Elemento que se muestra/oculta frecuentemente[hidden]Más rápido mostrar/ocultar que crear/destruir
Formulario con validaciones*ngIfLos inputs destruidos no afectan la validación

*ngFor - Repetir elementos

*ngFor recorre un array y crea un elemento HTML por cada item. Es como un bucle for pero en el HTML.

📄 *ngFor - Ejemplo con conversaciones
<!-- Por cada conversación en el array, crea un <li> --> <li *ngFor="let conv of conversations"> {{ conv.title }} </li> <!-- ¿Cómo se lee? --> <!-- "Para cada elemento (let conv) del array (of conversations)..." --> <!-- "...creá un <li> y mostrá conv.title" -->
📄 El array en el TypeScript
// En el .ts tenés el array: conversations = [ { id: 'conv-1', title: 'Planificación' }, { id: 'conv-2', title: 'Preguntas REST' }, { id: 'conv-3', title: 'Notas viejas' } ]; // El *ngFor crea 3 <li>: // <li>Planificación</li> // <li>Preguntas REST</li> // <li>Notas viejas</li>
💡 Analogía: *ngIf es como un interruptor de luz: la luz (el elemento) está encendida o apagada. *ngFor es como una fotocopiadora: toma un original (el template) y saca copias, una por cada item del array.

🎨 CAPÍTULO 7: ngClass, ngStyle Y PIPES

Ya sabés mostrar listas y ocultar elementos. Ahora vamos a aprender a cambiar el estilo visual de los elementos dinámicamente según el estado de la aplicación. Para eso existen ngClass, ngStyle y los Pipes.

ngClass - Clases CSS dinámicas

ngClass permite agregar o quitar clases CSS según condiciones del TypeScript. Es como decirle a Angular: "Ponele esta clase al elemento si esta condición es verdadera".

📄 ngClass - Ejemplos
<!-- ─── UNA CLASE CONDICIONAL ─── --> <div [class.active]="isActive">...</div> <!-- Si isActive = true → <div class="active"> --> <!-- Si isActive = false → <div class=""> --> <!-- ─── MÚLTIPLES CLASES ─── --> <div [ngClass]="{ 'active': conv.id === selectedId, 'unread': conv.hasUnreadMessages, 'archived': conv.status === 'ARCHIVED' }"> {{ conv.title }} </div> <!-- Se aplican solo las clases cuya condición es true -->

En UTNito, usamos [class.active] para resaltar la conversación seleccionada en el sidebar. Cuando el usuario hace click en una conversación, le ponemos la clase active y se ve diferente (con un fondo más claro y un borde).

ngStyle - Estilos inline dinámicos

ngStyle funciona igual que ngClass pero para estilos CSS inline. Permite cambiar propiedades CSS directamente desde el TypeScript.

📄 ngStyle - Ejemplos
<!-- Cambiar color de fondo según estado --> <div [ngStyle]="{ 'background-color': isActive ? '#cba6f7' : 'transparent', 'font-weight': hasUnread ? 'bold' : 'normal' }"> {{ conv.title }} </div> <!-- El operador ternario ? : decide qué valor usar -->

Pipes - Transformar datos antes de mostrarlos

Los Pipes son herramientas que transforman un valor antes de mostrarlo en pantalla. No modifican el dato original, solo cambian cómo se ve.

Pipe¿Qué hace?EjemploResultado
uppercaseConvierte a mayúsculas{{ 'hola' | uppercase }}HOLA
lowercaseConvierte a minúsculas{{ 'HOLA' | lowercase }}hola
titlecasePrimera letra mayúscula{{ 'hola mundo' | titlecase }}Hola Mundo
dateFormatea fechas{{ fecha | date:'short' }}4/30/26, 2:30 PM
jsonMuestra objeto como JSON{{ objeto | json }}{"id": 1, ...}

🔄 CAPÍTULO 8: CICLO DE VIDA DE UN COMPONENTE

Todo componente Angular tiene un ciclo de vida: nace, vive (se actualiza) y muere. Angular nos da hooks (ganchos) que son métodos que se ejecutan automáticamente en cada etapa. Podemos "engancharnos" a esos momentos para ejecutar nuestro propio código.

CICLO DE VIDA DE UN COMPONENTE 1. constructor() → Se crea la instancia. NO tiene @Input() todavía. 2. ngOnChanges() → Se ejecuta cuando cambia un @Input(). 3. ngOnInit() → Se ejecuta UNA SOLA VEZ. El componente está listo. 4. ngDoCheck() → Se ejecuta en cada verificación de cambios. 5. ngAfterViewInit() → La vista y sus hijos están listos. 6. ngOnDestroy() → El componente va a ser destruido. ¡Limpiar!

Los hooks más importantes

Hook¿Cuándo se ejecuta?¿Para qué sirve?Ejemplo en UTNito
constructorAl crear la instanciaInyectar servicios. ⚠️ NO llamar APIs acáconstructor(private fb: FormBuilder)
ngOnInitUna vez, después del constructor✅ Cargar datos del backend. El mejor lugarthis.chatService.loadConversations()
ngOnChangesCuando cambia un @Input()Reaccionar a cambios de propiedadesCuando cambia conversationId
ngOnDestroyAntes de destruir el componente✅ Cancelar subscripciones, limpiar timersthis.subscription.unsubscribe()

ngOnInit - El momento de cargar datos

ngOnInit es el hook más usado. Se ejecuta una sola vez, después del constructor, cuando el componente ya está completamente inicializado. Es el lugar ideal para:

  • ✅ Llamar a APIs y cargar datos del backend
  • ✅ Inicializar variables que dependen de @Input()
  • ✅ Configurar subscripciones a servicios
📄 Ejemplo de ngOnInit en UTNito
export class LoginComponent implements OnInit { // └────────────┘ // implements = "prometo que voy a tener ngOnInit" ngOnInit(): void { // Verificar si el usuario ya está logueado if (this.authService.isLoggedIn()) { this.router.navigate(['/chat']); } // Si ya está logueado, lo mandamos directo al chat } }

ngOnDestroy - Limpiar antes de irse

Cuando un componente se destruye (porque el usuario navega a otra página), hay que limpiar todo lo que quedó pendiente. Si no lo hacés, se producen memory leaks (pérdidas de memoria): el componente ya no existe pero sigue "escuchando" cosas.

📄 Ejemplo de ngOnDestroy
export class ChatComponent implements OnDestroy { private subscription: Subscription; ngOnInit(): void { // Nos suscribimos para recibir datos this.subscription = this.chatService.loadConversations().subscribe(); } ngOnDestroy(): void { // ⚠️ SIEMPRE cancelar subscripciones if (this.subscription) { this.subscription.unsubscribe(); } } }
⚠️ Cuidado con las subscripciones: Si no cancelás las subscripciones en ngOnDestroy, el componente sigue "escuchando" aunque ya no exista. Esto puede hacer que tu app se vuelva lenta con el tiempo. Siempre cancelá lo que suscribiste.

🔌 CAPÍTULO 10: SERVICIOS E INYECCIÓN DE DEPENDENCIAS

Hasta ahora toda la lógica de nuestra aplicación vive en los componentes. Pero en una aplicación real, los componentes deberían ocuparse solo de mostrar datos, no de obtenerlos ni procesarlos. Para eso existen los servicios.

¿Qué es un servicio?

Un servicio es una clase TypeScript que contiene la lógica de negocio y las llamadas al backend. Los componentes la usan, pero no la contienen. Esto mantiene el código organizado y reutilizable.

┌─────────────────────────────────────────────────────────┐ │ RESPONSABILIDADES │ │ │ │ COMPONENTE SERVICIO │ │ ┌──────────────────┐ ┌──────────────────┐ │ │ │ Mostrar datos │ │ Llamar al backend │ │ │ │ Escuchar eventos │ USA │ Transformar datos │ │ │ │ Validar UI │─────────►│ Manejar errores │ │ │ │ Navegar │ │ Compartir estado │ │ │ └──────────────────┘ └──────────────────┘ │ │ │ │ El componente MUESTRA. El servicio RESUELVE. │ └─────────────────────────────────────────────────────────┘

Analogía: El componente es como un mozo de restaurante (muestra el menú, toma el pedido, trae la comida). El servicio es la cocina (prepara la comida, habla con los proveedores, maneja los ingredientes). El mozo no cocina, y la cocina no atiende al cliente.

¿Qué es la Inyección de Dependencias?

En lugar de que el componente cree el servicio (con new), Angular se lo da ya creado. Esto se llama Inyección de Dependencias (DI). El componente declara qué necesita en su constructor y Angular se lo proporciona automáticamente.

📄 Inyección de Dependencias - Ejemplo
// ❌ SIN inyección de dependencias (NO se hace así en Angular) export class ChatComponent { constructor() { // El componente crea el servicio manualmente this.chatService = new ChatService(); // ❌ MAL } } // ✅ CON inyección de dependencias (ASÍ se hace en Angular) export class ChatComponent { // Angular CREA el servicio y nos lo DA constructor(private chatService: ChatService) {} // └───────────┘ └─────────┘ // Nombre local Tipo (la clase del servicio) }
💡 ¿Por qué es mejor la inyección de dependencias?
1. No repetís código: no hacés new en cada componente
2. Es más fácil de testear: podés reemplazar el servicio real por uno falso
3. Angular maneja el ciclo de vida: crea una sola instancia (singleton) y la comparte

@Injectable - Marcando una clase como servicio

El decorador @Injectable le dice a Angular: "Esta clase se puede inyectar en otros componentes o servicios".

📄 Estructura de un servicio
import { Injectable } from '@angular/core'; @Injectable({ providedIn: 'root' // ← Disponible en TODA la aplicación (singleton) }) export class AuthService { // Lógica de autenticación login(username: string, password: string): boolean { // ... validar credenciales ... return true; } }

providedIn: 'root' significa que el servicio se crea una sola vez para toda la aplicación (es un singleton). Todos los componentes que lo inyecten reciben la misma instancia.

¿Qué es HttpClient?

HttpClient es un servicio de Angular que permite hacer peticiones HTTP (GET, POST, PUT, PATCH, DELETE) al backend. Es como un cartero digital: lleva mensajes (requests) y trae respuestas.

En UTNito, tenemos un servicio base que contiene la lógica común para todas las llamadas HTTP. Los demás servicios (AuthApiService, ChatApiService) heredan de él.

import { HttpClient } from '@angular/common/http'; import { Injectable } from '@angular/core'; import { Observable } from 'rxjs'; import { environment } from '../../../environments/environment'; @Injectable({ providedIn: 'root' }) export class BaseApiService { // La URL base del backend (http://localhost:5001) protected apiUrl = environment.coreServiceUrl; // Inyectamos HttpClient constructor(protected readonly http: HttpClient) {} // Método genérico para GET protected get<T>(path: string): Observable<ResponseObject<T>> { return this.http.get<ResponseObject<T>>(`${this.apiUrl}/${path}`); // └───────┘ └──┘ // HttpClient Verbo GET } // Método genérico para POST protected post<T>(path: string, body: unknown): Observable<ResponseObject<T>> { return this.http.post<ResponseObject<T>>(`${this.apiUrl}/${path}`, body); // └───────┘ └───┘ └───┘ // HttpClient Verbo POST Datos que enviamos } // Método genérico para PATCH protected patch<T>(path: string, body: unknown): Observable<ResponseObject<T>> { return this.http.patch<ResponseObject<T>>(`${this.apiUrl}/${path}`, body); } }

Explicación línea por línea:

protected apiUrl = environment.coreServiceUrl → la URL del backend viene de una variable de entorno (http://localhost:5001)

protected readonly http: HttpClient → inyectamos HttpClient. protected permite que las clases hijas lo usen

<T> → tipo genérico. Permite que el método funcione con cualquier tipo de dato

`${this.apiUrl}/${path}` → template string. Si path es 'conversations', la URL queda http://localhost:5001/conversations

🔄 CAPÍTULO 12: OBSERVABLES Y RXJS

En el capítulo anterior vimos que HttpClient devuelve un Observable. Pero ¿qué es exactamente un Observable y por qué Angular lo usa en lugar de Promesas?

¿Qué es un Observable?

Un Observable es un flujo de datos en el tiempo. Es como una cañería por la que pueden pasar múltiples valores a lo largo del tiempo. A diferencia de una Promesa (que resuelve una sola vez), un Observable puede emitir varios valores, cancelarse y transformarse.

PROMESA (Promise) ──────────────────────────────────────────────► tiempo │ └── Resuelve UNA VEZ y termina OBSERVABLE ──────────────────────────────────────────────► tiempo │ │ │ │ └── Valor 1 │ │ └── Valor 2 │ └── Valor 3 └── Complete (terminó)

Las 3 cosas que puede hacer un Observable

Notificación¿Qué significa?¿Cuándo ocurre?
nextLlegó un nuevo valorCada vez que el backend responde o un evento ocurre
errorAlgo fallóError de red, backend caído, datos inválidos
completeTerminó el flujoCuando ya no hay más valores que recibir

subscribe() - La pieza más importante

subscribe() es lo que activa el Observable. Sin subscribe(), el Observable no hace nada. Es como abrir la canilla: el agua (los datos) no fluye hasta que alguien la abre.

📄 subscribe() - Estructura completa
this.chatApiService.listConversations() .subscribe({ next: (conversations) => { // Se ejecuta cuando llegan los datos correctamente console.log('Datos recibidos:', conversations); this.chatService.loadConversations(conversations); this.isLoading = false; }, error: (error) => { // Se ejecuta si algo falla (red, servidor, etc.) console.error('Error:', error); this.errorMessage = 'No se pudieron cargar los datos.'; this.isLoading = false; }, complete: () => { // Se ejecuta cuando el Observable termina (opcional) console.log('Carga completada'); } });
💡 Analogía: Un Observable es como una suscripción a un canal de YouTube. El canal (el backend) emite videos (datos). Vos te suscribís (subscribe) y recibís una notificación cada vez que hay un video nuevo (next). Si el canal cierra (error) o deja de publicar (complete), también te enterás.

Operadores RxJS: pipe, map, tap

RxJS (Reactive Extensions for JavaScript) es la librería que Angular usa para manejar Observables. Incluye operadores que permiten transformar, filtrar y combinar flujos de datos.

Operador¿Qué hace?Ejemplo en UTNito
pipe()Agrupa operadores para transformar datosEncadenar transformaciones
map()Transforma cada valor emitidoConvertir datos del backend al formato del frontend
tap()Ejecuta un efecto secundario sin modificar los datosGuardar en caché, mostrar logs
catchError()Atrapa errores y permite recuperarseMostrar mensaje de error amigable
📄 pipe() con map() y tap()

AuthApiService.login() - Conectando con el backend real

Veamos cómo AuthApiService usa pipe() y map() para extraer los datos de la respuesta del backend:

📄 auth-api.service.ts

💡 ¿Por qué extraemos response.data?
El backend siempre responde con la estructura ResponseObject<T>:
{ success, responseMessage, serverTime, data }

AuthApiService.login() extrae response.data para que AuthService reciba directamente el AuthSession sin tener que preocuparse por la estructura envolvente.

Esto mantiene la separación de responsabilidades: el servicio API se encarga de "desempaquetar" la respuesta del backend, y el servicio de negocio trabaja con datos limpios.

catchError - El operador que atrapa errores

catchError es un operador RxJS que intercepta errores en el flujo del Observable. Permite reaccionar al error y decidir qué hacer: mostrar un mensaje, reintentar, o devolver un valor por defecto.

Así como Angular organiza las pantallas con rutas, el backend de UTNito (NestJS) también organiza sus endpoints con una estructura de rutas anidadas:

Fijate que para eliminar un mensaje, la URL incluye ambos IDs: :id de la conversación y :mId del mensaje. El backend necesita saber a qué conversación pertenece el mensaje que queremos borrar.

El backend organiza los mensajes por conversación. La URL refleja esa jerarquía:

Ya tenés las rutas definidas. Ahora necesitás navegar entre ellas. Por ejemplo, después de un login exitoso, querés mandar al usuario al chat. Angular ofrece dos formas de navegar: desde TypeScript (con Router) y desde HTML (con routerLink).

Navegación desde TypeScript: Router.navigate()

El servicio Router se inyecta en el constructor y tiene un método navigate() que recibe un array con los segmentos de la URL.

📄 Navegación con Router.navigate()

¿Por qué navigate recibe un array? Cada elemento del array es un segmento de la URL. Angular los une con /. Esto permite construir rutas dinámicas fácilmente.

📄 Ejemplos de navigate()

🔒 CAPÍTULO 16: GUARDS - PROTEGIENDO RUTAS

En el capítulo anterior agregamos navegación entre login y chat. Pero hay un problema: si alguien escribe /chat directamente en el navegador, puede ver el chat sin loguearse. Para evitar esto existen los Guards (guardias).

¿Qué es un Guard?

Un Guard es una clase que decide si se puede acceder a una ruta. Es como un guardia de seguridad en la puerta de un edificio: te pregunta quién sos y si tenés permiso para entrar. Si no tenés permiso, te manda de vuelta.

USUARIO │ │ Escribe /chat en el navegador ▼ ┌──────────────────────────────────────────────────────────────┐ │ AUTH GUARD │ │ │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ ¿Está logueado? │ │ │ │ │ │ │ │ ✅ SÍ → Pase al ChatComponent │ │ │ │ ❌ NO → Redirigir al LoginComponent │ │ │ └─────────────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────────────┘

Implementando un Guard en Angular

Un Guard es una clase que implementa la interfaz CanActivate y tiene un método canActivate() que devuelve true (permitir acceso) o false (denegar acceso).

📄 Estructura de un Guard
import { Injectable } from '@angular/core'; import { CanActivate, Router } from '@angular/router'; // └─────────┘ // Interfaz que define un Guard. // Todo Guard debe implementarla. @Injectable({ providedIn: 'root' }) export class AuthGuard implements CanActivate { constructor(private router: Router) {} canActivate(): boolean { // ─── VERIFICAR SI HAY TOKEN ─── const token = localStorage.getItem('token'); // └────────────────────┘ // localStorage guarda datos en el navegador // que persisten aunque cierres la pestaña if (token) { return true; // ✅ Tiene token, puede pasar } // ❌ No tiene token, redirigir al login this.router.navigate(['/login']); return false; } }

AuthGuard en UTNito usando AuthService

En el proyecto real de UTNito no usamos localStorage con token. En su lugar, el Guard consulta al AuthService si hay un usuario logueado.

📄 AuthGuard de UTNito (con AuthService)
import { Injectable } from '@angular/core'; import { CanActivate, Router } from '@angular/router'; import { AuthService } from '../service/auth.service'; @Injectable({ providedIn: 'root' }) export class AuthGuard implements CanActivate { constructor( private router: Router, private authService: AuthService // ← Inyectamos nuestro servicio ) {} canActivate(): boolean { if (this.authService.isLoggedIn()) { return true; // ✅ Está logueado } this.router.navigate(['/login']); return false; // ❌ No está logueado } }

Usando el Guard en las rutas

El Guard se agrega en la definición de la ruta, en la propiedad canActivate.

📄 app.routes.ts con Guard
import { AuthGuard } from './guards/auth.guard'; export const routes: Routes = [ { path: 'login', component: LoginComponent }, // Ruta PROTEGIDA: solo usuarios logueados { path: 'chat', component: ChatComponent, canActivate: [AuthGuard] // ← El Guard protege esta ruta }, { path: '', redirectTo: '/login', pathMatch: 'full' }, { path: '**', redirectTo: '/login' }, ];
💡 Analogía: Un Guard es como el guardia de un boliche. Te pregunta si estás en la lista (logueado). Si estás, pasás. Si no, te manda de vuelta a la calle (login). El Guard se ejecuta antes de que el componente se cree, ahorrando recursos.

🗄️ CAPÍTULO 17: ESTADO EN SERVICIOS

Hasta ahora cada componente maneja sus propios datos. Pero en una aplicación real, varios componentes necesitan compartir la misma información. Por ejemplo: el ChatComponent y el SidebarComponent necesitan saber cuál es la conversación activa. Para eso usamos servicios con estado.

¿Qué es el estado?

El estado es toda la información que la aplicación necesita recordar en un momento dado. Por ejemplo:

┌──────────────────────────────────────────────────────────────┐ │ ESTADO DE UTNito │ │ │ │ AuthService ChatService │ │ ┌────────────────────┐ ┌────────────────────────────────┐ │ │ │ currentUser │ │ selectedConversationId │ │ │ │ (¿quién está │ │ (¿qué conversación está │ │ │ │ logueado?) │ │ seleccionada?) │ │ │ └────────────────────┘ │ │ │ │ │ conversations[] │ │ │ │ (lista de conversaciones) │ │ │ │ │ │ │ │ draftMessage │ │ │ │ (texto que está escribiendo) │ │ │ └────────────────────────────────┘ │ └──────────────────────────────────────────────────────────────┘

El problema: estado duplicado

Si cada componente guarda su propia copia de los datos, pueden quedar desincronizados. El SidebarComponent podría mostrar una conversación como activa mientras que el MessagesPanelComponent muestra otra.

❌ SIN estado centralizado (problema) SidebarComponent MessagesPanelComponent ┌──────────────────┐ ┌──────────────────────┐ │ activeId: conv-1 │ │ activeId: conv-2 │ └──────────────────┘ └──────────────────────┘ ⬆ ⬆ DIFERENTES ← ¡ERROR! ✅ CON estado centralizado (solución) ChatService ┌──────────────────┐ │ activeId: conv-1 │ ← UNA sola fuente de verdad └──────────────────┘ ⬇ ⬇ SidebarComponent MessagesPanelComponent (lee conv-1) (lee conv-1)

ChatService como fuente de verdad

En UTNito, el ChatService es el dueño de los datos de conversaciones. Los componentes solo leen y modifican a través de sus métodos. Esto se llama Single Source of Truth (una sola fuente de verdad).

📄 ChatService - Estado centralizado
@Injectable({ providedIn: 'root' }) export class ChatService { // ─── ESTADO PRIVADO (solo el servicio lo modifica) ─── private selectedConversationId: string | null = null; private conversations: Conversation[] = []; private draftMessage = ''; // ─── GETTERS (solo lectura) ─── getSelectedConversationId(): string | null { return this.selectedConversationId; } getConversations(): Conversation[] { return this.conversations; } getDraftMessage(): string { return this.draftMessage; } // ─── SETTERS (única forma de modificar el estado) ─── selectConversation(conversationId: string): void { this.selectedConversationId = conversationId; console.log('Conversación seleccionada:', conversationId); } setDraftMessage(value: string): void { this.draftMessage = value; } }

¿Por qué los datos son privados y se accede con getters/setters? Porque así controlamos cómo se modifican. Si un componente pudiera hacer chatService.selectedConversationId = 'otra-cosa' directamente, podríamos tener bugs difíciles de rastrear. Con un método selectConversation(), podemos agregar logs, validaciones, o efectos secundarios.

💡 Analogía: El servicio con estado es como una biblioteca. Los libros (datos) están en un solo lugar. Los lectores (componentes) pueden consultar qué libros hay (getters) y pedir prestado (setters), pero no pueden modificar los libros directamente. La bibliotecaria (el servicio) controla todo.

🔗 CAPÍTULO 18: @INPUT Y @OUTPUT - COMUNICACIÓN ENTRE COMPONENTES

Ya sabés que los servicios permiten compartir datos entre componentes. Pero hay otra forma de comunicación más directa: pasar datos de un componente padre a un componente hijo (@Input) y enviar eventos del hijo al padre (@Output).

El flujo de datos en Angular

PADRE (ChatComponent) │ │ @Input() ← pasa datos al hijo ▼ HIJO (SidebarComponent) │ │ @Output() ← emite eventos al padre ▼ PADRE (ChatComponent) recibe el evento

@Input() - Pasar datos del padre al hijo

📋 CAPÍTULO 19: INTERFACES Y MODELOS TYPESCRIPT

En los capítulos anteriores usamos tipos como string, boolean y Conversation[]. Pero ¿de dónde sale el tipo Conversation? De una interfaz TypeScript. Las interfaces definen la forma que deben tener los datos en tu aplicación.

¿Qué es una interfaz?

Una interfaz es un contrato que dice qué propiedades y tipos debe tener un objeto. No genera código JavaScript, solo ayuda a TypeScript a verificar que los datos tengan la forma correcta mientras escribís.

📄 Ejemplo de interfaz
// Así definimos la forma de una Conversación export interface Conversation { id: string; // Todas las conversaciones tienen un ID de texto title: string; // Todas las conversaciones tienen un título archived: boolean; // Todas las conversaciones saben si están archivadas messages: Message[]; // Todas las conversaciones tienen un array de mensajes } // Ahora TypeScript sabe que esto ES una Conversation: const conv: Conversation = { id: 'conv-1', title: 'Planificación', archived: false, messages: [] }; // Y esto NO es una Conversation (falta 'messages'): const invalido: Conversation = { id: 'conv-1', title: 'Planificación' }; // ❌ ERROR: Property 'messages' is missing

No genera código JavaScriptSí genera código JavaScript
Solo existe en tiempo de compilaciónExiste en tiempo de ejecución
Se usa para tipar objetos planosSe usa para lógica de negocio (servicios, componentes)
Ej: interface Conversation { ... }Ej: class ChatService { ... }

🔧 CAPÍTULO 20: ENUMS Y VARIABLES DE ENTORNO

En el capítulo anterior creamos interfaces. Ahora vamos a ver dos herramientas más que hacen nuestro código más seguro y flexible: los Enums (para valores fijos) y las variables de entorno (para configuraciones que cambian según dónde se ejecute la app).

¿Qué es un Enum?

Un Enum (enumeración) es un tipo de dato que solo puede tener valores predefinidos. Es como un menú de opciones: solo podés elegir una de las opciones disponibles, nada más.

📄 Sin Enum vs Con Enum
// ❌ SIN Enum: cualquier string es válido (propenso a errores) let role: string = 'user'; role = 'admin'; // ✅ Válido role = 'cualquier'; // ✅ Válido... ¿pero es correcto? role = 'usr'; // ✅ Válido... pero debería ser 'user' // ✅ CON Enum: solo valores definidos (seguro) enum MessageRole { USER = 'user', // Único valor permitido para usuario ASSISTANT = 'assistant', // Único valor permitido para asistente } let role: MessageRole = MessageRole.USER; role = MessageRole.ASSISTANT; // ✅ Válido role = 'admin'; // ❌ ERROR: no es un valor del Enum

MessageRole - El Enum de UTNito

En UTNito, cada mensaje tiene un role que indica quién lo envió: el USER (humano) o el ASSISTANT (bot). Usamos un Enum para que no haya errores de tipeo.

📄 message-role.enum.ts
export enum MessageRole { USER = 'user', // Mensaje del humano ASSISTANT = 'assistant', // Respuesta del bot } // Uso en el componente: import { MessageRole } from './message-role.enum'; // En el HTML podemos usar el Enum para clases condicionales: <!-- [class.user]="message.role === messageRole.USER" -->

¿Qué son las variables de entorno?

Las variables de entorno son configuraciones que cambian según dónde se ejecute la aplicación. Por ejemplo, en desarrollo el backend está en localhost:5001, pero en producción estaría en miapp.com/api.

📄 environment.ts (desarrollo)
export const environment = { production: false, // No estamos en producción chatTitle: 'UTNito', // Título de la app chatAgentVersion: 'v1', // Versión del asistente coreServiceUrl: 'http://localhost:5001', // URL del backend en desarrollo routeLogin: 'login', // Ruta del login routeChat: 'chat', // Ruta del chat };
📄 Cómo se usa en BaseApiService
import { environment } from '../../../environments/environment'; export class BaseApiService { protected apiUrl = environment.coreServiceUrl; // └────────────────────┘ // Lee la URL desde la configuración // En desarrollo: http://localhost:5001 // En producción: https://api.utnito.com }
💡 Analogía: Un Enum es como un semáforo: solo puede estar en rojo, amarillo o verde. No existe "semáforo en azul". Las variables de entorno son como la ropa: usás ropa diferente según dónde estés (desarrollo = pantalones cortos en casa, producción = traje en la oficina).
